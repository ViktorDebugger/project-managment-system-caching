# Сценарій тестування та знімання скріншотів для звіту

## Передумови

- Запущена система: `docker-compose up --build` (або `docker compose up --build`)
- Усі сервіси в стані `Up`, Redis і БД — `healthy`
- Swagger доступний за адресою http://localhost:3000/api

---

## Етап 1. Перевірка запуску контейнерів

### Команда

```bash
docker ps
```

або

```bash
docker compose ps
```

### Дія

Зробити скріншот виводу зі списком контейнерів (user_db, workspace_db, project_task_db, redis, user-auth, workspace, project-task, api-gateway).

### Підпис до скріншота

> **Рис. 1. Контейнери Docker.** Усі сервіси (БД, Redis, мікросервіси, API Gateway) у стані Up.

---

## Етап 2. Реєстрація та отримання JWT

### Дія

1. Відкрити Swagger: http://localhost:3000/api
2. Розгорнути `POST /auth/register`
3. Виконати запит з тілом:

```json
{
  "username": "testuser",
  "email": "test@example.com",
  "fullname": "Test User",
  "password": "Test123!"
}
```

4. Розгорнути `POST /auth/login`
5. Виконати запит з тілом:

```json
{
  "email": "test@example.com",
  "password": "Test123!"
}
```

6. Скопіювати значення `access_token` з відповіді

### Скріншот 1

Знімок екрану з успішним відповіддю `POST /auth/login` та видимим `access_token`.

### Підпис

> **Рис. 2. Отримання JWT.** Успішна автентифікація та отримання токена доступу.

---

## Етап 3. Авторизація в Swagger та створення workspace

### Дія

1. Натиснути кнопку **Authorize** в Swagger
2. Ввести `Bearer <ваш_access_token>`
3. Підтвердити авторизацію
4. Викликати `POST /workspaces` з тілом:

```json
{
  "name": "Test Workspace",
  "description": "Для тестування кешу"
}
```

5. Зберегти `id` створеного workspace з відповіді (наприклад, `95363c99-5000-4351-849a-1f2c5637ad86`)

### Скріншот 2

Знімок з успішним створенням workspace та видимим `id` у відповіді.

### Підпис

> **Рис. 3. Створення workspace.** Результат запиту `POST /workspaces` з ідентифікатором створеного робочого простору.

---

## Етап 4. Перший виклик GET (без кешу)

### Команда (через curl у терміналі)

```bash
curl -w "\nЧас відповіді: %{time_total}s\n" -s -o /dev/null -H "Authorization: Bearer ВАШ_TOKEN" http://localhost:3000/workspaces/ВАШ_WORKSPACE_ID
```

Замінити `ВАШ_TOKEN` та `ВАШ_WORKSPACE_ID` на реальні значення.

### Альтернатива — через Swagger

1. Викликати `GET /workspaces/{workspaceId}` з `workspaceId` з попереднього кроку
2. Зняти скріншот з відповіддю та заголовками/часом виконання (якщо є)

### Скріншот 3

Знімок терміналу з результатом `curl`, де видно рядок з часом (наприклад, `Час відповіді: 0.052s`).

### Підпис

> **Рис. 4. Перший виклик GET /workspaces/:id (без кешу).** Час відповіді ~50–100 ms, дані беруться з БД.

---

## Етап 5. Другий виклик GET (з кешу)

### Команда

Та сама, що в етапі 4, виконати ще раз:

```bash
curl -w "\nЧас відповіді: %{time_total}s\n" -s -o /dev/null -H "Authorization: Bearer ВАШ_TOKEN" http://localhost:3000/workspaces/ВАШ_WORKSPACE_ID
```

### Скріншот 4

Знімок терміналу з другим викликом та меншим часом (наприклад, `Час відповіді: 0.008s`).

### Підпис

> **Рис. 5. Повторний виклик GET /workspaces/:id (з кешу).** Час відповіді ~5–15 ms, дані повертаються з Redis.

---

## Етап 6. Порівняння часу відповіді

### Дія

Підготувати коротку таблицю результатів:

| Виклик | Джерело | Час (приблизно) |
|--------|---------|-----------------|
| 1-й (без кешу) | БД | ___ ms |
| 2-й (з кешу) | Redis | ___ ms |

Можна зробити один скріншот з двома командами та їхнім виводом (етапи 4 і 5).

### Підпис

> **Рис. 6. Порівняння часу відповіді.** Відповідь з кешу значно швидша порівняно з запитом до БД.

---

## Етап 7. Інвалідація кешу при оновленні

### Дія

1. Викликати `PUT /workspaces/{workspaceId}` з тілом:

```json
{
  "name": "Test Workspace Updated"
}
```

2. Після оновлення знову викликати `GET /workspaces/{workspaceId}` — перший після оновлення виклик буде без кешу (MISS), наступні — з кешу (HIT).

### Скріншот 5 (опційно)

Знімок Swagger з результатом оновлення workspace.

### Підпис

> **Рис. 7. Оновлення workspace та інвалідація кешу.** Після PUT кеш очищається, наступний GET знову йде до БД.

---

## Етап 8. Перезапуск контейнерів

### Команда

```bash
docker compose restart workspace
```

або

```bash
docker-compose restart workspace
```

### Дія

1. Виконати перезапуск
2. Дочекатися, поки сервіс знову стане здоровим
3. Зняти скріншот стану контейнерів:

```bash
docker ps
```

### Підпис

> **Рис. 8. Перезапуск контейнера workspace.** Система залишається робочою після перезапуску окремого сервісу.

---

## Етап 9. Логи workspace (опційно)

### Команда

```bash
docker logs project-managment-workspace-1 --tail 30
```

Можна додати тимчасове логування кешу в код (як було раніше), щоб у логах були рядки `[Cache] GET workspace:... MISS` та `[Cache] GET workspace:... HIT`. Якщо логування прибрано — скріншот звичайних логів workspace достатній.

### Підпис

> **Рис. 9. Логи сервісу workspace.** Приклад виводу під час роботи з кешем.

---

## Мінімальний набір скріншотів для звіту

1. **Рис. 1** — `docker ps` (усі контейнери)
2. **Рис. 2** — отримання JWT (login)
3. **Рис. 3** — створення workspace
4. **Рис. 4** — перший GET (час без кешу)
5. **Рис. 5** — другий GET (час з кешу)
6. **Рис. 6** — порівняння часу (або об'єднаний скріншот з етапів 4–5)
7. **Рис. 7** — оновлення workspace (опційно)
8. **Рис. 8** — перезапуск контейнера та `docker ps`

---

## Примітка щодо Spring Cache / @Cacheable

У проєкті використовується NestJS, а не Java Spring. Аналог Spring Cache та `@Cacheable`:

- `@nestjs/cache-manager` + Redis
- Ручне кешування через `CACHE_MANAGER` (get/set/del) у сервісі

У звіті можна описати це як еквівалент Spring Cache в екосистемі Node.js/NestJS.
